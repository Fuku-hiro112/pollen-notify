2. 真実の基準（DDL）

src/main/resources/db/migration/V1__init.sql が DB スキーマの真実。Entity はこれに 1:1 で対応させる。ddl-auto: validate モードのため、不一致は起動失敗で検知される。

observation_points
├─ id                BIGSERIAL    PRIMARY KEY
├─ external_id       VARCHAR(64)  NOT NULL UNIQUE   ← API の citycode を入れる
├─ name              VARCHAR(128) NOT NULL
├─ latitude          NUMERIC(9,6)
├─ longitude         NUMERIC(9,6)
├─ created_at        TIMESTAMPTZ  NOT NULL DEFAULT NOW()
└─ updated_at        TIMESTAMPTZ  NOT NULL DEFAULT NOW()

pollen_measurements
├─ id                    BIGSERIAL   PRIMARY KEY
├─ observation_point_id  BIGINT      NOT NULL  FK → observation_points(id)
├─ measured_at           TIMESTAMPTZ NOT NULL
├─ pollen_count          INTEGER     NOT NULL
├─ created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
└─ UNIQUE(observation_point_id, measured_at)

3. ObservationPoint.java 修正チェックリスト

対象：src/main/java/com/fuku/pollen_notify/entity/ObservationPoint.java

  • import 追加：@Entity, @Table, @GeneratedValue, @SequenceGenerator, @Column 等を jakarta.persistence.* から
  • @Table：@Table(name="Pollen") → @Table(name = "observation_points")
  • フィールドを DDL に合わせる：country / prefecture / city を撤去し、以下を追加：
    • externalId（String, API の citycode を入れる, NOT NULL, UNIQUE）
    • name（String, 地点名）
    • latitude（BigDecimal, @Column(precision=9, scale=6)）
    • longitude（BigDecimal, 同上）
    • createdAt（Instant）
    • updatedAt（Instant）
  • id の型：Integer → Long（BIGSERIAL 対応）
  • 採番戦略：@GeneratedValue(strategy = SEQUENCE) + @SequenceGenerator を追加
    • sequenceName = "observation_points_id_seq"
    • allocationSize = 1（罠回避）
  • メソッド命名：GetId / SetId → getId / setId（JavaBeans 規約。大文字始まりだと JPA / Lombok が認識しない）
  • setter 撤去：方針通り不変性を保つため、すべての setXxx を削除。状態変更が要るなら意味のあるメソッドへ（例：updateLocation(lat, lon)）
  • created_at / updated_at は Java 側でセットしない：DDL の DEFAULT NOW() に委譲。getterだけ用意
💡 setterを撤去した後、テストで状態を作りたい時はどうする？

ファクトリメソッドを public static で用意する。例：

public static ObservationPoint create(String externalId, String name,
                                       BigDecimal lat, BigDecimal lon) {
    ObservationPoint p = new ObservationPoint();
    // 内部でフィールドを埋める（同パッケージから見える設計）
    return p;
}

これで「正しい状態のものしか作れない」を保証できる。

4. PollenMeasurement.java 実装チェックリスト

対象：src/main/java/com/fuku/pollen_notify/entity/PollenMeasurement.java（現状空クラス）

  • クラスアノテーション：@Entity + @Table(name = "pollen_measurements")
  • Lombok（採用方針）：@Getter, @NoArgsConstructor(access = PROTECTED), @EqualsAndHashCode(of = "id")
  • id：Long 型に @Id + @GeneratedValue(SEQUENCE) + @SequenceGenerator
    • sequenceName = "pollen_measurements_id_seq"
    • allocationSize = 1
  • observationPoint（参照）：
    • @ManyToOne(fetch = FetchType.LAZY, optional = false)
    • @JoinColumn(name = "observation_point_id", nullable = false)
    • 型は ObservationPoint
    • ⚠ 逆方向の @OneToMany は持たせない（地点に観測値リストを持たせない）
  • measuredAt：Instant, NOT NULL（観測時刻＝絶対時刻）
  • pollenCount：int または Integer。NOT NULL 想定なら int でも可
    • ※ -9999（欠測）はそもそも保存しない方針 → Entity に来る時点で正の整数
  • createdAt：Instant、DDL の DEFAULT NOW() に委譲
  • ファクトリメソッド検討：
    • public static PollenMeasurement of(ObservationPoint p, Instant at, int count) {...}
    • この中で「count が負数や -9999 ならエラー」のドメインルールを書ける
ファクトリメソッドにドメインルールを集中させる効果：
  • 「不正な状態の PollenMeasurement は作れない」を仕組みで保証できる
  • Service層が「-9999 を除外する処理」を持つ必要がなくなる（責務分散の解消）
  • Phase 4 のドメイン層強化への布石

5. 検証手順（end-to-end）

  1. docker compose ps で DB が Up (healthy) を確認（落ちていたら docker compose up -d）
  2. ./gradlew bootRun
  3. 起動成功 = ddl-auto: validate がスキーマ一致を保証してくれた状態
  4. 失敗時：Schema-validation: missing column ... 等のエラー全文を確認し、DDL と Entity を照合
  5. 起動できたら Step 2（Repository）へ進む
