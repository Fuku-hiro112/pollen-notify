1. ./gradlew とは何か

gradlew は Gradle Wrapper（グレイドル・ラッパー）と呼ばれるファイルで、プロジェクトのルート（一番上の階層）に置かれています。

OS使うファイル打ち方の例
macOS / Linuxgradlew./gradlew compileJava
Windows（あなた）gradlew.bat.\gradlew compileJava または gradlew compileJava
Windowsでの注意： ネット上の記事は Mac 前提で ./gradlew と書かれていることが多いです。Windows の PowerShell では ./gradlew でも動きますが、迷ったら .\gradlew（バックスラッシュ）で打つのが確実です。先頭の .\ は「今いるフォルダの中の」という意味です。
💡 なぜ Wrapper を使うのか？
あなたのPCに Gradle を別途インストールしていなくても、gradlew が「このプロジェクト用の正しいバージョンのGradle」を自動で用意して実行してくれます。チーム全員が同じバージョンでビルドできる＝「私のPCでは動くのに…」を防ぐ仕組みです。

compileJava って何をするコマンド？

「Javaのソースコードだけをコンパイル（機械が読める形に変換）して、文法エラーがないか確かめる」タスクです。アプリの起動まではしません。「今書いたコードが文法的に正しいか」を素早く確認するのにちょうど良い、軽いコマンドです。

コマンドやること使う場面
gradlew compileJava本体コードのコンパイルのみ書いたコードのエラー確認（今これ）
gradlew buildコンパイル＋テスト＋成果物作成全部まとめて検証したいとき
gradlew bootRunSpring Boot アプリを起動実際にアプリを動かすとき（Phase後半）

2. コマンドを打つ場所は3つある

「どこで打てばいい？」の答えは 「次の3つのどれでもOK」 です。やることは同じですが、おすすめ順に並べます。

共通の大前提：必ず「プロジェクトのルート」で打つこと。
ルートとは gradlew.bat や build.gradle.kts が直接置いてあるフォルダ＝ C:\dev\ClaudeProject\pollen-notify\pollen-notify\ です。
別のフォルダ（例：src の中）で打つと 「gradlew が見つからない」エラーになります。方法A・Bは IntelliJ が自動でルートを基準にしてくれるので、この点でつまずきません。だから初心者には A・B がおすすめです。

3. IntelliJ の画面構成

まず「どこに何があるか」を地図として頭に入れます。IntelliJ の画面は大きく4つの領域に分かれています。

💡 ツールウィンドウとは： IntelliJ では、画面の端から出てくるパネル（Project, Terminal, Gradle など）をツールウィンドウと呼びます。普段は隠れていて、端のボタンを押すと出てきます。「見当たらない」ときは閉じているだけなので、端のバーを探してください。

4. 方法A：内蔵ターミナルで打つ（おすすめ）

コードを見ながら同じウィンドウ内でコマンドを打てるので、これが一番ラクです。

1 IntelliJ で pollen-notify プロジェクトを開いている状態にする。
（もし開いていなければ：上部メニュー File → Open… → C:\dev\ClaudeProject\pollen-notify\pollen-notify フォルダを選ぶ）

2 内蔵ターミナルを開く。次のどちらかで開けます：

  • 画面下端（または左下）の Terminal という文字をクリック
  • ショートカット Alt + F12
  • 見つからなければ：上部メニュー View → Tool Windows → Terminal

3 開いたターミナルの行頭が、すでにプロジェクトのルートになっているはずです。例：

PS C:\dev\ClaudeProject\pollen-notify\pollen-notify>

この PS は PowerShell の印。末尾が pollen-notify> なら正しい場所です。

4 コマンドを打って Enter：

.\gradlew compileJava

成功すると BUILD SUCCESSFUL、失敗すると BUILD FAILED とエラー内容が表示されます。

初回は時間がかかります。 一番最初の実行では Gradle 本体や依存ライブラリのダウンロードが走るため、数十秒〜数分かかることがあります。2回目以降は速くなります。固まったように見えても、しばらく待ってください。

5. 方法B：Gradle ツールウィンドウから実行（コマンド不要）

「コマンドを覚えたくない／打ち間違えが不安」という場合は、クリックだけで実行できます。

1 画面右端の Gradle（象のアイコン🐘）をクリックして Gradle ツールウィンドウを開く。
（無ければ View → Tool Windows → Gradle）

2 ツリーを次のようにたどる：

pollen-notify
└─ Tasks
   └─ build
      └─ compileJava   ← これをダブルクリック

3 ダブルクリックすると自動で実行され、下部の Build / Run ウィンドウに結果が出ます。

💡 Tasks が表示されないとき： Gradle ツールウィンドウ上部の歯車から「Gradleタスクを一覧表示」設定が要る場合があります。面倒なら方法A（ターミナル）の方が確実です。まずは方法Aを基本にして大丈夫。

6. ビルドエラーの読み方

エラーは怖がらず「ファイル名・行番号・メッセージ」の3点を読みます。たとえば今日あなたが遭遇したエラー：

ObservationPoint.java:22: エラー: シンボルを見つけられません
@NoArgsConstructor(access = PROTECTED)
                            ^  シンボル: 変数 PROTECTED
読む場所意味
ObservationPoint.javaどのファイルか
:2222行目（IntelliJでこの数字をクリックすると該当行へ飛べる）
シンボルを見つけられませんその名前が定義されていない／importされていない
^ の位置問題のある列を指す矢印
💡 IntelliJ なら打つ前に気づける： エディタ上でエラー箇所には赤い波線が引かれます。マウスを乗せると理由がポップアップし、Alt + Enter で「import を追加」などの修正候補が出ます。ターミナルでビルドする前に、まず赤い波線を見る癖をつけると速いです。
ただし注意： Lombok のようにアノテーション処理が絡むエラーは、IDEの表示とコマンドのビルド結果が食い違うことがあります。最終的な正解は gradlew compileJava の結果だと考えてください（実際のコンパイラが下す判定だからです）。

7. 覚えておくと速いショートカット（Windows）

ショートカット機能
Shift ×2（連打）何でも検索（ファイル・クラス・設定）。一番便利
Alt + F12ターミナルを開く/閉じる
Alt + Enterエラー箇所で修正候補を出す（import追加など）
Ctrl + S保存（※IntelliJは自動保存だが、ビルド前に押すと安心）
Ctrl + Alt + Lコード整形（インデントを自動で揃える）
Ctrl + B定義へジャンプ（メソッドやクラスの中身を見る）
Shift + F10直前の構成を実行（アプリ起動など、Phase後半で使う）

8. 理解度チェック

Q1. gradlew compileJava は、プロジェクトのどのフォルダで打つべき？

A. gradlew.bat / build.gradle.kts が直接置いてあるルートフォルダ（...\pollen-notify\pollen-notify\）。方法A・Bを使えば IntelliJ が自動でこの場所を基準にしてくれる。

Q2. compileJava と build の違いは？

A. compileJava はコンパイル（文法チェック）だけ。build はコンパイル＋テスト＋成果物作成まで全部やる。今は「書いたコードの文法確認」が目的なので軽い compileJava で十分。

Q3. エディタに赤い波線が無いのに gradlew compileJava が失敗した。どちらを信じる？

A. コマンドの結果（実際のコンパイラの判定）を信じる。特に Lombok などアノテーション処理が絡むと、IDE表示とコンパイル結果がズレることがある。

Q4. Windowsで ./gradlew と .\gradlew、どちらが確実？

A. PowerShellではどちらも動くが、Windowsの流儀は .\gradlew（バックスラッシュ）。迷ったらこちら。先頭の .\ は「今いるフォルダの中の」という意味。

9. 次にやること

  1. 残り1個のコンパイルエラーを直す：ObservationPoint.java:22 の @NoArgsConstructor(access = PROTECTED) を、AccessLevel.PROTECTED と書く（A案）か、import static lombok.AccessLevel.PROTECTED; を足す（B案）。
    ヒント：すでに 17行目で import static jakarta.persistence.GenerationType.SEQUENCE; と同じことをやっている。
  2. 方法A（内蔵ターミナル）で .\gradlew compileJava を打ち、BUILD SUCCESSFUL を自分の目で確認する。
  3. 通ったら、PollenMeasurement.java も同じ要領でコンパイルを通す。
🔭 この先（Phase 4への伏線）： 今は「コンパイルが通ること」がゴールですが、Phase 4 のDDDリファクタでは、この ObservationPoint を集約の単位として捉え直し、「どうやって正しく生成・変更させるか（ファクトリ・不変条件）」を設計し直します。今 @NoArgsConstructor(access = PROTECTED) で「外から勝手に空インスタンスを作らせない」としているのは、その伏線です。
