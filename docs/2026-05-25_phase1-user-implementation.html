1Step 1: Entity を書く

学習対象： JPA アノテーション、ORM マッピング、不変オブジェクト の設計

作るもの

  • PollenMeasurement.java — pollen_measurements テーブルに対応

調べてほしいアノテーション

アノテーション役割（自分で調べる）
@EntityこのクラスはJPA管理対象
@Table(name=...)テーブル名を明示
@Id主キーフィールド
@GeneratedValue(strategy=...)自動採番。IDENTITY vs SEQUENCE の違いを調べると良い
@Column(name=..., nullable=..., unique=...)カラム属性
@ManyToOne / @JoinColumnFK関連（PollenMeasurement → ObservationPoint）
@UniqueConstraint複合UNIQUE（uq_measurementに対応）

設計判断ポイント（自分で決める）

① ID生成戦略：IDENTITY か SEQUENCE か？

テーブル定義は BIGSERIAL ＝ 裏で SEQUENCE が動く形式。JPA側で GenerationType.IDENTITY と SEQUENCE どちらでも動くが、挙動が違う。「永続化のタイミングでINSERT発行 vs 事前にSEQUENCE取得」など。

調べてみるキーワード：「JPA IDENTITY SEQUENCE 違い」「Hibernate batch insert」

② 日時型：LocalDateTime / Instant / OffsetDateTime どれを使うか？

テーブル定義は TIMESTAMPTZ（タイムゾーン付き）。LocalDateTime はタイムゾーン情報を持たないので不適合。Instant か OffsetDateTime が候補。

調べてみるキーワード：「JPA Instant OffsetDateTime TIMESTAMPTZ」

③ 関連の持ち方：双方向 or 単方向？

PollenMeasurement から ObservationPoint への参照は確実に必要（@ManyToOne）。逆方向（ObservationPoint が List<PollenMeasurement> を持つ）は持たせない方が無難（遅延ロードの沼にはまりがち）。

調べてみるキーワード：「JPA 単方向 双方向 アンチパターン」

④ Lombok使う？使わない？

build.gradle.kts には既に Lombok が入っている。@Getter, @NoArgsConstructor(access = PROTECTED) 等が定番。ただし JPA との相性で @Data や @EqualsAndHashCode を雑に使うと事故る。

調べてみるキーワード：「JPA Lombok アンチパターン」

⚠ やりがちな失敗
  • @Data を Entity に貼る → equals/hashCode が全フィールド比較になり、双方向関連で 無限ループ。
  • 引数なしコンストラクタを public で公開 → ドメイン的に不正な状態のEntityが作れてしまう。protected 推奨。
  • ID生成戦略を AUTO にする → ベンダー依存で予期せぬ挙動になる。明示すべき。

動作確認方法

  1. Entityを書く
  2. ./gradlew bootRun
  3. 起動成功すれば ddl-auto: validate がスキーマ一致を保証してくれる
  4. 起動失敗時は Schema-validation: missing column 等のエラーが出る → Entity定義とDDLを照合

📖 公式： Spring Data JPA Reference / Jakarta Persistence Spec

Phase 4 では ObservationPoint を 集約ルート に昇格、pollen_count や measured_at を 値オブジェクト に切り出します。

4. 理解度チェック

Q1. なぜ Entity と DTO を分けるのか？

答え： 外部APIの仕様変更（カラム追加・型変更）が永続化層に直接波及するのを防ぐため。境界で受けるDTOが盾になる。Phase 4 のDDDで言う「腐敗防止層（Anti-Corruption Layer）」の入り口。

Q2. @ManyToOne のデフォルトのフェッチ戦略は EAGER か LAZY か？

答え： EAGER（即時ロード）。これが意外と落とし穴。多くの実務では LAZY に変えるのが定石。@ManyToOne(fetch = FetchType.LAZY) と明示する。

Q3. なぜ pollen_count の欠測値 -9999 をスキップするのか？DBに保存して後で除外ではダメか？

答え： 「観測値テーブルの整合性」と「ドメインの意味」を一致させるため。pollen_count は「実測された花粉数」と意味づけしたなら、欠測はそもそも入れないのが正しい。Phase 4 で値オブジェクトに昇格させた時、「pollen_count は非負整数」というルールが綺麗に書ける。

Q4. @Transactional は Service と Repository どちらに付けるべきか？

答え： 通常は Service（ユースケース境界）。Repository に付けるとトランザクション境界が細かくなりすぎ、複数Repository呼出が別トランザクションになり整合性が崩れる。

Q5. Phase 1 で「動く貧弱なコード」を作る理由は？

答え： CLAUDE.md にもある通り、まず動くものを作って全体の流れを掴むため。設計を最初から考えすぎると進まない。Phase 4 でリファクタする題材になる。
