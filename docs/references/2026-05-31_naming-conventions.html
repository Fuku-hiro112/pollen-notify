なぜ命名規則を決めるのか
命名は「未来の自分と読み手への手紙」です。規則がバラバラだと、created_at なのか createdAt なのかを毎回SQLとコードで照合する羽目になります（実際このプロジェクトで列名ミスのバグが起きました）。「どの層では何ケースを使うか」を一度決めて機械的に従うと、考える量が減り、バグも減ります。

1. まず4つの命名ケースを覚える

プログラミングの命名は、ほぼこの4種類の「区切り方」の組み合わせです。

ケース名見た目区切り方主にどこで使う
PascalCase
（アッパーキャメル）		ObservationPoint各単語の頭を大文字、区切りなしクラス名・型名
camelCase
（ローワーキャメル）		pollenCount先頭だけ小文字、以降の単語頭は大文字変数・メソッド・フィールド
snake_casepollen_count全部小文字、単語を _ で区切るDBのテーブル・列名
UPPER_SNAKE_CASE
（スクリーミングスネーク）		NO_DATA全部大文字、_ 区切り定数（static final）
🟣 覚え方：Java側はキャメル系、DB側はスネーク系。この「世界が違う」ことが次章の変換の話につながります。

3. 最重要：Java ↔ DB の命名変換

ここがこのプロジェクトで一番事故った場所です。Javaのフィールドは camelCase、DBの列は snake_case。この橋渡しを JPA / Hibernate が自動でやってくれます。

🔵 仕組み：Hibernate の既定の命名戦略（PhysicalNamingStrategy）が、externalId → external_id のようにキャメル→スネークへ自動変換します。だから @Column(name=...) を書かなくても正しい列名に対応づきます。
🔴 このプロジェクトで実際に起きたバグ：@Column(name = "createdAt") と手書きでキャメルのまま指定 → DBの実列名は created_at なので不一致 → 起動/クエリでエラー。
教訓：列名は「書かない（自動変換に任せる）」のが一番安全。もし書くなら必ず snake_case で、SQLの実列名と一字一句合わせる。

5. ファクトリメソッドの命名

論点⑥で of(...) を採用しましたが、静的ファクトリには意味で使い分ける慣習名があります（JDK標準ライブラリの命名にならうと読み手に伝わりやすい）。

名前意味・使う場面JDKの例
of(...)引数の値からそのまま作る（最も汎用）List.of() Optional.of()
from(...)別の型から変換して作るInstant.from() LocalDate.from()
valueOf(...)文字列/プリミティブから変換Integer.valueOf()
create() / newXxx()毎回新しいインスタンスを作ることを強調—
getInstance()同じインスタンスを返しうる（キャッシュ/シングルトン）Calendar.getInstance()
🟣 Phase 4 への伏線：DDDでは「ファクトリ」が値オブジェクトや集約を正しく組み立てる役になります。今の of() に書いた「負数なら例外」のガード節が、その不変条件チェックの原型です。

7. やりがちな失敗（Phase 1）

🔴 失敗1：引数名に _ 接頭辞
of(ObservationPoint _observationPoint, ...) のように _ を付けるのはC#などの流儀で、Javaでは非推奨。Javaは普通に observationPoint でよい（フィールドと同名でも、代入時は this.observationPoint = observationPoint; で区別できる）。
→ あなたは既に _ を外して修正済み 👍
🔴 失敗2：列名を手書きでキャメルのまま
@Column(name = "createdAt") → DBは created_at なので不一致。書かないのが安全。書くなら snake_case。
🟡 注意：パッケージ名の pollen_notify
Javaのパッケージは本来「全小文字・アンダースコアなし」が理想（pollennotify）。ただし Spring Initializr がプロジェクト名から自動生成したもので、今から直す必要はない（実害なし）。「本当はこうあるべき」だけ知っておけばOK。
🟡 注意：略語は避ける
cnt obsPt のような独自略語は読み手に伝わらない。count observationPoint と書く。ただし id url api など広く通じる略語はOK。

8. 理解度チェック

Q1. DBの列が measured_at のとき、Javaのフィールド名は何にすべき？ @Column(name=...) は必要？

measuredAt（camelCase）。Hibernateが measuredAt → measured_at に自動変換するので @Column(name=...) は不要。書かないのが安全。

Q2. -9999 を定数にするとき、正しい命名はどれ？ noData / NoData / NO_DATA

NO_DATA。定数（static final）は UPPER_SNAKE_CASE。noData は変数、NoData はクラスの見た目になってしまう。

Q3. テーブル名は単数形 observation_point と複数形 observation_points どちら？

このプロジェクトは複数形（observation_points）で統一。「1行=1地点、テーブルは地点の集まり」という考え方。※単数派の流儀もあるが、プロジェクト内で統一が最重要。

Q4. 静的ファクトリ of と from の使い分けは？

of は引数の値からそのまま組み立てる汎用。from は別の型から変換して作る場合（例：APIのDTOから PollenMeasurement.from(dto)）。Phase 1後半でDTO変換を書くとき from の出番が来るかも。

Q5. なぜDBは created_at、Javaは createdAt と「世界で違う」のに困らない？

Hibernate（JPA実装）の命名戦略が両者を自動で橋渡しするから。プログラマはそれぞれの世界の慣習に従って書くだけでよい。手書きで @Column(name) を混ぜると、この自動変換と食い違ってバグる。
