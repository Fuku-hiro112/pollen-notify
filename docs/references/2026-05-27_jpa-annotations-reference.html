1. @Entity 必須

用途

クラスを JPA 管理対象（=テーブルに対応するクラス）として宣言する。

シグネチャ

@Entity(name = "...")

name 属性は省略可（省略時はクラス名がそのままエンティティ名になる）。

使用例

@Entity
public class ObservationPoint {
    // ...
}

メリット / デメリット

観点評価
付けるとSpring Data JPAが自動でCRUD対象として認識Repository から型指定するだけで使える
JPA仕様の制約を負う引数なしコンストラクタ必須、finalクラス不可など

本プロジェクトでの推奨

@Entity のみ。name属性は付けない（クラス名 = エンティティ名で十分）。

📖 Jakarta Persistence Spec - Entity

2. @Table 推奨

用途

Entityがマップされるテーブル名・スキーマ・複合UNIQUE制約を指定する。

シグネチャ

@Table(name = "...", schema = "...", uniqueConstraints = {...}, indexes = {...})

使用例

@Entity
@Table(name = "observation_points")
public class ObservationPoint { ... }

複合UNIQUE制約の指定例（PollenMeasurement の uq_measurementに対応）：

@Entity
@Table(name = "pollen_measurements",
       uniqueConstraints = @UniqueConstraint(
           name = "uq_measurement",
           columnNames = {"observation_point_id", "measured_at"}
       ))
public class PollenMeasurement { ... }

メリット / デメリット

観点説明
テーブル名がスネークケース、クラス名がパスカルケース、という変換ルールから外れても明示できる本プロジェクトは ObservationPoint → observation_points と複数形＋スネークケース。Hibernateの自動変換でも一致する可能性はあるが、明示的な方が誤解を防ぐ。
UNIQUE制約を Entity 側に書けるDDL（Flyway）と二重管理になるが、Entity定義を読むだけでテーブルの不変条件が分かる利点
DDL二重管理真実は Flyway 側。Entity側のuniqueConstraintsはddl-auto: validateで検証されない（あくまでcreate/update時の生成用）

本プロジェクトでの推奨

  • @Table(name = "...") は必ず付ける（テーブル名の明示）
  • uniqueConstraints は付けても付けなくてもOK。付ける場合は「ドキュメント目的」と割り切る。実際の制約は Flyway の V1__init.sql 側で定義済

3. @Id 必須

用途

主キーフィールドを指定する。Entityには必ず1つ必要。

使用例

@Id
private Long id;

注意点

  • 型は Long 推奨（プリミティブ long ではない）。理由：JPAは「未永続化＝idがnull」で判定するため、nullになれる必要がある
  • 複合主キーは @IdClass または @EmbeddedId で表現。本プロジェクトでは代理キー単独なので不要

本プロジェクトでの推奨

private Long id; に @Id を付ける。両Entity共通。

4. @GeneratedValue 必須

用途

主キー値を誰がいつ採番するかを指定する。

シグネチャ

@GeneratedValue(strategy = GenerationType.XXX, generator = "...")

主要な strategy

strategy動作本プロジェクト適合
IDENTITYDBのauto_increment / SERIAL に委譲。INSERT時に採番△ batch insertが効かない
SEQUENCEDBのSEQUENCEオブジェクトを使う。INSERT前に番号取得✅ 採用
TABLE採番用テーブルを使う。レガシー❌ 使わない
AUTOベンダー任せ❌ 明示性なし。避ける
UUIDJava側でUUID生成（Hibernate 6.2+）❌ 今回は連番ID

使用例（SEQUENCE採用）

@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE,
                generator = "observation_points_id_seq")
@SequenceGenerator(name = "observation_points_id_seq",
                   sequenceName = "observation_points_id_seq",
                   allocationSize = 1)
private Long id;

メリット / デメリット（SEQUENCE視点）

観点評価
事前にIDが確定するequals(of="id") 戦略と矛盾しない（未保存でも比較可能）
batch insertが効く大量INSERTの性能向上
SEQUENCE名がDBと一致してないと起動失敗BIGSERIALは <テーブル名>_<カラム名>_seq の命名でSEQUENCEが自動作成される
allocationSize 不一致で番号飛び 罠下の @SequenceGenerator 項参照

本プロジェクトでの推奨

strategy = GenerationType.SEQUENCE + 後述の @SequenceGenerator を併用。

5. @SequenceGenerator SEQUENCE採用時必須

用途

JPA側でSEQUENCEの詳細（名前、確保サイズ）を定義する。

シグネチャ

@SequenceGenerator(name = "...", sequenceName = "...", allocationSize = N, initialValue = N)
属性意味
nameJPA内部での識別子。@GeneratedValue(generator=...)と一致させる
sequenceNameDB上の実SEQUENCE名。BIGSERIALが裏で作るSEQUENCEの名前と一致必須
allocationSize1回のnextval()で何個ID予約するか
initialValue初期値（既にDB側で決まっていれば指定不要）

BIGSERIAL が裏で作るSEQUENCE名

テーブル observation_points でカラム id を BIGSERIAL にした場合、PostgreSQLが自動で作るSEQUENCE名は：

observation_points_id_seq

確認コマンド：

docker compose exec db psql -U pollen -d pollen_notify -c "\ds"

allocationSize の罠 注意

Hibernate のデフォルト allocationSize は 50。これは「Hibernate が DB の SEQUENCE を 1回 nextval して、その値から 50 個分を自分で割り当てる」という最適化。

一方、BIGSERIAL が作る SEQUENCE のデフォルト INCREMENT は 1。

この2つが不一致だと：

  • Hibernate: nextval() で 1 を取得 → 自分で 1〜50 を割り当てる気でいる
  • DB側: 次の nextval() で 2 を返す（Hibernateの予約 2〜50 と衝突）
  • 結果：重複ID または 番号飛び

解決策2択：

  1. (推奨) JPA側を allocationSize = 1 にする：性能は落ちるが安全。学習段階はこれ
  2. DB側のSEQUENCEを ALTER SEQUENCE ... INCREMENT BY 50 して JPA も allocationSize = 50 にする：性能◎だが二重管理

本プロジェクトでの推奨

allocationSize = 1 でスタート。性能が必要になったらDDLと一緒に変更。

6. @Column 推奨

用途

カラム名・NULL許可・UNIQUE・長さ・精度などを明示する。

シグネチャ（主要属性）

@Column(name = "...", nullable = false, unique = true, length = N, precision = N, scale = N)

使用例

@Column(name = "external_id", nullable = false, unique = true, length = 64)
private String externalId;

@Column(nullable = false, length = 128)
private String name;

@Column(precision = 9, scale = 6)
private BigDecimal latitude;

@Column(name = "created_at", nullable = false)
private Instant createdAt;

主要属性のメリット / デメリット

属性使う理由注意
nameJavaフィールド名（camelCase）→ DBカラム名（snake_case）の対応を明示Hibernateの自動命名規則でも一致する場合は省略可。だが明示が安全
nullableDDLとEntityの整合を表現ddl-auto: validateでは nullable は検証されない ことに注意（生成時にしか効かない）
unique単一カラムUNIQUE複合UNIQUEは @Table.uniqueConstraints へ
lengthVARCHAR長の明示DDLと不一致でも validate ではエラーにならない（型の長さは検査されない）
precision / scaleNUMERIC型の精度BigDecimal使用時に必須級
誤解されがちな点： @Column の属性の多くは「Hibernateがテーブルを生成する時のヒント」であり、ddl-auto: validate モードではカラムの存在と型カテゴリくらいしか検証されません。「nullable = false と書いたから安心」ではなく、真の制約は Flyway の DDL 側です。

本プロジェクトでの推奨

  • フィールド名と DB カラム名が一致しない場合（snake_case変換が発生）は name を明示
  • BigDecimalフィールドには precision, scale を必ず指定
  • nullable はドキュメント目的で書いてOK（読み手への意図伝達）

7. @ManyToOne FK必須

用途

「多 : 1」の関連を表現する。PollenMeasurement（多） → ObservationPoint（1）が今回のケース。

シグネチャ

@ManyToOne(fetch = FetchType.LAZY, optional = false, cascade = {})

使用例

@ManyToOne(fetch = FetchType.LAZY, optional = false)
@JoinColumn(name = "observation_point_id", nullable = false)
private ObservationPoint observationPoint;

主要属性

属性意味本プロジェクト推奨
fetch関連の取得タイミング。LAZY（必要になったらSQL発行）/EAGER（即時取得）LAZY 必須。デフォルトはEAGERという罠あり
optional関連が必須か（false なら必須＝NULL不可）FKがNOT NULLなので false
cascade親操作の連鎖（例：保存時に関連も保存）使わない。明示制御の方が安全

メリット / デメリット

観点説明
関連先オブジェクトをJavaで素直に扱えるmeasurement.getObservationPoint().getName() で参照可
EAGERの罠@ManyToOne はデフォルトEAGER。LAZYを明示しないと、Measurementを1件取るたびにObservationPointも自動取得 → N+1問題の温床
LazyInitializationExceptionトランザクション外で measurement.getObservationPoint() を呼ぶと例外。Service内（@Transactional内）で必要な情報を取り切る必要
⚠ 双方向にしない
ObservationPoint 側に List<PollenMeasurement> を持たせる（@OneToMany(mappedBy=...)）誘惑があるが、避ける。理由：
  • 地点の観測値は何万件にもなる → 全件ロードしたら破滅
  • 「地点を起点に観測値を取りたい」なら PollenMeasurementRepository.findByObservationPoint(...) を書けば良い
  • 双方向は同期メソッドが必要になり複雑化

本プロジェクトでの推奨

@ManyToOne(fetch = LAZY, optional = false) + @JoinColumn。逆方向の @OneToMany は持たせない。

8. @JoinColumn 推奨

用途

FK列の名前・NULL許可などを指定する。@ManyToOneとセットで使う。

シグネチャ

@JoinColumn(name = "...", nullable = false, referencedColumnName = "...")

使用例

@JoinColumn(name = "observation_point_id", nullable = false)

メリット / デメリット

観点説明
FK列名を明示できるJava側フィールド名は observationPoint、DB側カラム名は observation_point_id という命名差を埋める
読み手にDB構造が伝わるEntity定義を読むだけでテーブル間の関係が見える
省略すると Hibernate のデフォルト命名規則で推測される意図しない名前になり validate で失敗することがある

本プロジェクトでの推奨

必ず @JoinColumn(name = "observation_point_id", nullable = false) を @ManyToOne の直下に書く。

9. @UniqueConstraint（@Table 内で使う） DDL一致用

用途

複合UNIQUE制約をJPA側で宣言する。単一カラムなら @Column(unique=true) で済むので、これは「2カラム以上のUNIQUE」用。

使用例

@Table(name = "pollen_measurements",
       uniqueConstraints = @UniqueConstraint(
           name = "uq_measurement",
           columnNames = {"observation_point_id", "measured_at"}
       ))
public class PollenMeasurement { ... }

メリット / デメリット

観点説明
Entity定義を読めば「この組み合わせは一意」と分かるドキュメント効果
validateモードでは検証されない実体はFlyway側のDDLが真実
DDL二重管理変更時にFlywayとEntity両方を直す必要

本プロジェクトでの推奨

書いてもよいが、必須ではない。書く場合は「読み手のためのドキュメント」と割り切る。

10. 補足：その他見かけるアノテーション

アノテーション用途本プロジェクト使用
@Version楽観的ロック用の版番号フィールドPhase 1 では不要。Phase 3の同時更新対策で検討
@Enumerated(EnumType.STRING)Java enumをVARCHARで保存今は使うフィールドなし
@Embedded / @Embeddable複合値の埋め込み（例：住所をクラスとして持ちつつ複数カラムに展開）Phase 4 で値オブジェクト化する時に登場予定
@CreationTimestamp / @UpdateTimestampHibernate拡張。created_at / updated_at を自動セット使わない。DDL側で DEFAULT NOW() 済みなので、Entity側は何もしなくて良い。むしろ二重管理になる
@PrePersist / @PreUpdate永続化前後のフックPhase 1では不要
@Transient「このフィールドはDBに保存しない」今は不要
created_at / updated_at の扱い注意
DDL側に DEFAULT NOW() を設定済み。つまりINSERT時に明示的に値を与えなくてもDBが入れてくれる。
Entity側で @CreationTimestamp を使ったり、コンストラクタで Instant.now() をセットしたりすると、2箇所で時刻決定の責務を持つことになり、わずかなずれが生まれる可能性。
推奨：Entity側では一切セットしない。getterだけ用意して、DB側に任せる。
Phase 4で 監査属性（audit fields）を「ドメイン的に意味のあるタイミングで自分で設定する」リファクタが入ることもあります（例：「観測時刻」と「DB登録時刻」は別物として明示）。

12. 理解度チェック

Q1. @ManyToOne のデフォルトのフェッチ戦略は？

答え： EAGER（即時取得）。明示的に fetch = FetchType.LAZY を指定しないと、N+1問題やパフォーマンス劣化の原因になる。

Q2. @Column(nullable = false) と書けば、テーブルに NOT NULL 制約が保証される？

答え： いいえ。nullable は Hibernate がテーブルを生成する時のヒントであり、ddl-auto: validate モードでは検証されない。真のNOT NULL制約は Flyway の DDL 側で定義される。

Q3. allocationSize の値が DB の SEQUENCE の INCREMENT と異なるとどうなる？

答え： Hibernate が予約したID範囲と、DBが次に返すIDが衝突して、重複ID や 番号飛び が発生する。両者は必ず一致させる必要がある。

Q4. なぜ ObservationPoint に List<PollenMeasurement> を持たせない方が良い？

答え： (1) 観測値は時系列で大量に蓄積するので全件ロードは破滅的、(2) 必要なら Repository 側で findByObservationPoint(...) を書けば取れる、(3) 双方向関連は同期メソッドを要するので複雑化する。

Q5. なぜ Entity の created_at を Java側でセットせず DB の DEFAULT NOW() に任せるべき？

答え： 時刻決定の責務が2箇所（Java側のクロックとDB側のクロック）に分散すると、わずかなずれや矛盾が生まれる。「真実は1箇所に」 という原則から、DB側に一元化する。Java側はgetterだけ用意して読み取り専用にする。
