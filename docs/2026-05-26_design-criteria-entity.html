1. 共通の判断軸（OOP・保守性視点）

3つの論点はそれぞれ独立に見えますが、「OOPに則り保守性高く」という目標から逆算すると、共通の判断軸が見えてきます。

あなたが「OOP・保守性」を重視するなら、特に ①カプセル化 / ②不変性 / ③同一性 / ⑦拡張安全性 の4軸を最重要視するのが筋です。⑤⑥は手段として効いてきます。⑧は Phase 4 のDDDで主役級になります。

2. 論点1: Lombokを使うか

Lombokとは： アノテーション処理器で @Getter, @Setter, @Builder, @RequiredArgsConstructor 等を書くと、コンパイル時にgetter等を自動生成する。「定型コードを書かなくて済む」がウリ。

選択肢

選択肢具体的アノテーション例
A. フル活用@Data, @Builder, @AllArgsConstructor
B. 安全な範囲だけ@Getter, @NoArgsConstructor(access=PROTECTED), @EqualsAndHashCode(of="id")
C. 使わない（手書き）全部自分で書く / Java record を使う（ただし @Entity と相性悪い）

判断軸ごとの評価

軸A: フル活用B: 安全範囲C: 手書き
① カプセル化❌ setterで内部状態が外から変更可能✅ setterを生成しない選択ができる✅ 完全制御
② 不変性❌ setterで破壊される✅ Getterのみ生成 → 実質不変✅ finalフィールドで強制可能
③ 同一性（equals/hashCode）❌ 全フィールド比較 → JPAで事故✅ IDのみで比較を明示✅ 自分で意図的に書く
⑤ 可読性△ 一見短いが裏で何が起きてるか不明△ ある程度✅ コードが事実
⑥ テスト容易性✅ Builder便利△ ファクトリメソッド自作必要△ 同上
⑦ 拡張安全性❌ 「setter」が誘発する破壊的変更が増える✅ Setterがない＝意図しない変更を防げる✅ 完全制御
⚠ JPA × Lombokの著名な事故パターン
  • @Data を Entity に貼る → equals/hashCode が全フィールド比較 → 双方向関連で無限ループ。
  • @AllArgsConstructor を Entity に貼る → IDも含めて引数になり、外部からID指定の生成が可能に。自然キー破壊。
  • @Builder を Entity に貼る → 同上。@Builder はDTOやValueObjectで使うべき。
  • @NoArgsConstructor を public で生やす → 不正な状態のEntityが作れる。JPAはprotectedでも反射で呼べる。

あなたの軸でのおすすめ

B案（安全範囲）が筋。 具体的には： 
@Entity
@Table(name = "observation_points")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)  // JPA用、外からは使えない
@EqualsAndHashCode(of = "id")                       // ID基準で同一性を明示
public class ObservationPoint {
    @Id @GeneratedValue(...) private Long id;
    private String externalId;
    // ...
    // setterは生成しない。状態変更は意味のあるメソッド経由
    // 例: public void updateLocation(BigDecimal lat, BigDecimal lon) { ... }
}

理由：「getterは公開しても害が少ない」「setterは公開すると point.setId(99) のような破壊的操作を許してしまう」という非対称性。getterのみLombok、書き換えは 意味のあるメソッド として手書きする。

もう一段の選択肢： 学習目的を重視するなら C案（手書き） も非常に良い。「Lombokが裏で何をしているか」を理解する近道になる。Phase 1だけ手書き、Phase 2以降はBに移行、も可。
Phase 4 で値オブジェクト（VO）を導入すると、VOには @Value（Lombokの不変オブジェクト用）が候補になります。EntityとVOで Lombok の使い分けが発生します。

📖 参考： Lombok公式 機能一覧 / JPA × equals/hashCode のベストプラクティス (Vlad Mihalcea)

3. 論点2: ID生成戦略

背景： DDLで BIGSERIAL（PostgreSQLの自動採番）にした。JPA側で対応する @GeneratedValue(strategy=?) の選択肢が複数ある。

選択肢

戦略動作
IDENTITYINSERT文を発行 → DBが採番した値をJavaに戻す。INSERTごとに通信1往復。
SEQUENCE事前に SELECT nextval() で番号を取得 → JavaがIDを持った状態でINSERT。事前取得＋INSERT。
AUTOベンダー任せ。PostgreSQLでは多くの場合SEQUENCEになるが明示性がない。
UUIDJava側でUUID生成。DBに番号生成を頼まない。分散環境向き。

判断軸ごとの評価

軸IDENTITYSEQUENCEUUID
③ 同一性△ 永続化までIDがnull✅ 事前にID確定✅ アプリ側でID確定
⑥ テスト容易性❌ save()するまでequals不能✅ ID付きで生成可能✅ 同上
⑦ 拡張安全性（性能含む）❌ Hibernateのbatch insertが効かない✅ batch可✅ batch可
⑧ ドメイン整合△ 整数連番△ 整数連番✅ グローバル一意
可読性（ログ・URL）✅ /points/1✅ /points/1❌ /points/a8f3...
学習対象としての面白さ△ 簡単✅ Hibernateの挙動が学べる△ 別軸の学び

あなたの軸でのおすすめ

SEQUENCE が筋。 理由：
  • ③同一性の明確さ：永続化前でもIDが確定する＝equals(of="id")戦略と矛盾しない。
  • ⑦拡張安全性：将来観測値を100件単位でINSERTする際、Hibernate の batch insert が効く（IDENTITYでは不可）。
  • ⑥テスト容易性：単体テストでEntityをsaveせずにequals検証できる。
  • 学習効果：「Hibernateは内部でどうSEQUENCEを使うか（allocationSize、pooled-loアルゴリズム）」を学ぶ題材になる。

UUIDはマイクロサービス・分散システムの題材になるが、Phase 1には過剰。学習目的ならPhase 5の拡張として温めておく。

SEQUENCE採用時のJPA記述の注意点 
@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "observation_points_id_seq")
@SequenceGenerator(name = "observation_points_id_seq",
                   sequenceName = "observation_points_id_seq",
                   allocationSize = 50)   // ←ここ重要
private Long id;

allocationSize は「1回のnextvalで何個番号を予約するか」。デフォルト50はHibernateのデフォルトでDB側のSEQUENCEのINCREMENTと不一致だと番号が飛ぶ。DBのSEQUENCE INCREMENT と一致させるのがセオリー。

BIGSERIAL は INCREMENT 1。なので最初は allocationSize = 1 でスタートし、性能が要るようになったらDDLとJPAを揃えて変更、が安全。

📖 参考： JPA Entity Identifier Generation (Vlad Mihalcea)

4. 論点3: 日時型

背景： DDLで TIMESTAMPTZ（タイムゾーン情報を持つ絶対時刻）にした。Javaの日時APIから何を選ぶか。

Javaの主要日時型

型意味例
LocalDateTimeタイムゾーンなしの日時。「2026-05-26 10:00」カレンダーの予定
OffsetDateTimeUTCからのオフセット付き日時。「2026-05-26 10:00+09:00」ログ・APIのI/O
ZonedDateTimeタイムゾーンID付き日時。「2026-05-26 10:00 Asia/Tokyo」夏時間考慮のスケジュール
InstantUTC基準の絶対時刻（タイムライン上の点）イベント発生時刻、システム内部処理

判断軸ごとの評価

軸LocalDateTimeInstantOffsetDateTime
① TIMESTAMPTZ との整合❌ TZ情報を持たない✅ UTC固定✅ オフセット保持
⑤ 可読性✅ 2026-05-26T10:00△ 2026-05-26T01:00Z（UTC表記でとっつきにくい）✅ 2026-05-26T10:00+09:00
⑥ テスト容易性✅ TZ気にしなくて良い✅ Clockでモック可能△ オフセット制御要
⑧ ドメイン整合（"観測時刻"の意味）❌ 「どこの10時」か曖昧✅ 絶対時刻＝観測の本質△ 観測地のオフセット情報も持つ
API I/O（ISO8601 with offset）との相性❌ 情報落ち△ 受信時にオフセット情報を捨てる✅ そのまま往復
JPA/Hibernate 公式サポート✅ 標準✅ 標準✅ 標準

あなたの軸でのおすすめ

判断は measured_at のドメイン的意味で決まる。

選択肢A：Instant（推奨）

「観測時刻」をタイムライン上の絶対点として捉える立場。
東京の10:00もNYの10:00も「別の点」として明確に区別される。観測という事象は世界に1つしかない点を持つ、というドメイン解釈と完全一致。

✅ OOP的に最強の不変性：余計な情報を持たない＝ドメイン本質に集中

選択肢B：OffsetDateTime

「観測時刻＋観測地点のオフセット」を一緒に持つ立場。
APIレスポンスがISO8601で+09:00を含むなら、そのまま受けて落とさず保存できる。「日本の地点なので+09:00で記録された」というメタ情報を保持できる。

✅ 情報を捨てない

❌ 観測地点の緯度経度から推測可能な情報をわざわざ持つ＝冗長性増

結論ロジック

「保守性」を重視するなら、データの冗長を避ける Instant が筋。「観測地点」は別エンティティ（ObservationPoint）が持つ責任なので、観測値が「自分はどこのTZだったか」を覚える必要はない（責務分離＝④単一責任）。

application.yml の設定との整合

既に hibernate.jdbc.time_zone: UTC を設定済。これは「Java側のLocalDateTimeをDBに渡す際UTCとみなす」設定で、Instant/OffsetDateTimeを使う場合は無関係に動く。混乱を防ぐため、EntityではLocalDateTimeを絶対に使わないと決めるのが吉。

Phase 4 では measured_at を MeasuredAt という値オブジェクトに昇格させる候補になります。その時、内部に Instant を持ち、「過去N分以内か」「営業時間中か」などのビジネスロジックを内包させます。

📖 参考： java.time パッケージ (Oracle公式)

6. 理解度チェック

Q1. なぜ Entity の equals/hashCode を「全フィールド比較」ではなく「ID基準」にすべきか？

答え： Entity の同一性（identity）は「DB上の同じレコードを指すか」で決まるべき。同じレコードなら、値が更新されても等価。逆に、別レコードなら全フィールド一致しても等価ではない。これがEntityの本質。値で同等性を判定したいオブジェクトは Entity ではなく Value Object として別途設計する。

Q2. IDENTITY 戦略では、なぜ Hibernate の batch insert が効かないのか？

答え： Hibernate は「INSERT文を複数まとめて投げて性能を稼ぐ」最適化を持つが、IDENTITY だと「1件INSERT→DBが返したIDを取得→次のINSERT」を順次行う必要があり、まとめられない。SEQUENCE なら事前にIDを確保できるので、まとめてINSERT可能。

Q3. なぜ LocalDateTime ではなく Instant を使うべきか？

答え： LocalDateTime はTZ情報を持たない＝「どこの10時か」が不明。観測値という事象は世界に1つの時点で起きたものなので、表現する型もそれと一致した「絶対時刻」型であるべき。Instant は UTCタイムライン上の点を表す唯一の型。

Q4. setter を生成しない代わりに、状態を変更したい時はどう書く？

答え： 「意味のあるメソッド」として手書きする。例：point.updateLocation(lat, lon), point.rename(newName)。setter（setLatitude）は「何のために値を変えるのか」を示せないが、メソッド名なら意図が表現できる。これはOOPの語彙（ユビキタス言語）の第一歩。

Q5. SEQUENCE の allocationSize を 50（Hibernateデフォルト）にして、DBの INCREMENT BY を 1 のままにすると何が起きる？

答え： Hibernate は「nextval()で取得した値から50個分は自分で割り当てる」と考える。一方DBは1ずつしかインクリメントしないので、Hibernate内部の予約番号がDBの実際の次番号と衝突 → 重複IDが採番される or 番号飛びが発生する。両者は必ず一致させる。「設定がDBとJPAで二重管理になっている」のは保守性の敵。
