# 花粉通知アプリ

## プロジェクト概要
指定した花粉飛散量がしきい値を超過した時に通知するアプリ。私生活で実用しつつ、Spring Boot + DDD/設計を学ぶための題材。

## 学習目標（最重要）
このプロジェクトの **第一目的は学習** であり、「動くアプリを最速で作ること」ではない。
- 重点習得：Spring Boot / JPA / REST API設計 / DDD / テスト / リファクタリング
- 「インプット2：アウトプット8」の比率を守る
- 自分で書き、間違え、修正するプロセスを経ることで学習する

## Claudeへの依頼方針（重要）

**Claudeはメンター / レビュアー / 設計相談役。実装の代行はしない。**

### 推奨される依頼
- 設計判断の選択肢提示（A案 vs B案 のトレードオフ説明）
- ユーザーが書いたコードのレビュー・改善提案
- エラーメッセージの解説と原因仮説
- 概念の説明（例：「JPAの遅延ロードとは」「集約境界の決め方」）
- テストケースの観点出し（実装はユーザー）
- 詰まった時の「ヒント」（答えではなく方向性）

### 避けるべき依頼
- 「〇〇機能を実装して」「全部書いて」「最後まで作って」
- 自分で考えずに丸投げする質問
- 答えだけを聞く質問

### 例外（Claudeが書いてOK）
- ボイラープレート（pom/build.gradle、application.yml、Initializr相当の雛形）
- 学習対象外の周辺コード（HTMLテンプレートの装飾、シェルスクリプト等）
- ユーザーが「これは自分で書く対象ではない」と明示した部分

### Claudeへの応答ルール
- 質問された時、コードを書く前に「これはユーザーが書くべき部分か？」を判断する
- 書くべきと判断したら、まずユーザーの現在の理解と試行を確認する
- すぐ完成形を出さず、ヒントから始める段階的支援を優先する

## 技術スタック
- 言語：Java 21
- フレームワーク：Spring Boot 3.x
- 永続化：Spring Data JPA + PostgreSQL（ローカルはDocker Compose）
- 通知：Discord Webhook（Phase 2）、LINE Messaging API（Phase 5以降の任意拡張）
- 外部API：ウェザーニューズ「ポールンロボ」花粉飛散数データ
- ビルド：Gradle (Kotlin DSL)
- テスト：JUnit 5 / Mockito / AssertJ
- バージョン管理：Git

## 開発フェーズ

| Phase | 目的 | 主な学習対象 |
|---|---|---|
| 1 | Spring Boot雛形 + 花粉API取得・DB保存 | Spring基礎、JPA、外部API連携 |
| 2 | 定期実行 + しきい値判定 + Discord通知 | バッチ、通知ポート抽象化 |
| 3 | ユーザー設定をREST APIで管理 | API設計、認証、テスト |
| 4 | DDDリファクタ（**本命**） | 集約、値オブジェクト、ドメインサービス |
| 5 | フロント or デプロイ（任意） | - |

各Phaseで「動く貧弱なコード」を一旦完成させてから次に進む。Phase 4でそれまでのコードを設計の本で学んだ概念で殴り直す。

## ドメイン用語集（ユビキタス言語）
開発を進めながら追記していく。

- **花粉観測値 (Pollen Measurement)**: ある観測地点・時刻の花粉飛散数（個/m³）
- **観測地点 (Observation Point)**: 市区町村単位の観測ポイント
- **通知設定 (Notification Preference)**: ユーザーが設定する通知条件（しきい値、時間帯、対象地域）
- **通知判定 (Notification Decision)**: 観測値 + 通知設定 から通知すべきか判断するドメインロジック
- **通知履歴 (Notification History)**: 過去送信した通知の記録（重複通知防止）

## 進め方のルール
1. 新機能は **まず動く最小実装** を作る。設計は後で考える
2. 詰まったら **最低1時間は自分で試す**。それからClaudeに「ヒント」を求める
3. Phase完了時に **学んだ点・ハマった点をメモ** する（将来のskill素材）
4. テストは Phase 3 以降で本格導入。Phase 1〜2は手動確認でも可
5. コミットは細かく。1コミット = 1論理的変更
